# Commit Issues - Post-MVP Report

A working companion to `MVP-Plan.md`, written after the first build-and-playtest cycle. The short version: the ten-level campaign is functionally complete and matches the plan closely, analytics ended up more thorough than originally scoped, a couple of the original design laws were knowingly bent to fix real confusion a real player hit, and the whole thing has never been run through an automated test - every fix so far has come from someone actually playing it and hitting a wall. That last point should shape how much confidence to place in what follows.

## What we covered from the plan

The design law mostly held. Zero friction to first challenge is intact - the terminal loads with level one already running, no account, no onboarding carousel. "Almost no body text" held too: each objective is still one to three lines styled as a comment. Dark-by-default with a single swappable accent color shipped as specified, with seven presets (green, amber, cyan, magenta, blue, red, white) against the plan's "6-8," persisted through `localStorage` exactly as planned.

The scope cut to campaign-only, ten levels, held exactly as written: stage a specific file, commit, create and switch branches, stash before a blocked checkout, pop that stash, a clean three-way merge, a real merge conflict resolved through an in-terminal editor, unstaging a file, a soft reset that keeps a change staged, and a revert on a commit that's already shared. Nothing from the "out" list crept back in - no Free Practice, no Blitz mode, no Daily Challenge, no cherry-pick/rebase/reflog, no leaderboard. That boundary held under its own weight; there was never a moment where a level needed a command outside the planned ~15-20 subcommand vocabulary.

The core loop matches the spec closely: objective as a comment, real command entry with tab-completion and up/down history, a real engine that returns actual Git-shaped output or errors rather than a canned "wrong answer," and a running timer. The three-tier hint system (nudge, command family, exact command) shipped as planned, and - better than planned - every hint request is now a trackable event rather than just a local counter. The success/error color convention from the visual spec (accent for success, a fixed muted red for errors regardless of theme) shipped exactly as written.

## Where we deviated from the plan

The most consequential change is the engine. The plan called for isomorphic-git plus an in-memory filesystem, run through a build step. The sandbox available for this build couldn't run Node or npm for the entire session, so instead of a real Git library, the engine is a hand-written model: full-file snapshots per commit, a real staging/workdir/HEAD split, real fast-forward vs. three-way merge logic, real conflict markers, and real soft/mixed/hard reset differences - but it is not literally Git. It enforces the semantics the levels need, and every level's solution path was traced by hand against the code rather than run, but it is a simplified stand-in, not the planned dependency. Worth revisiting if the project ever needs Git behavior beyond what the ten levels exercise.

The build ended up simpler than planned in one respect: there's no bundler at all, not even the Vite the plan suggested. It's five plain script files loaded in order with no build step, which arguably serves "zero friction" even better than planned, at the cost of no module system if the codebase grows.

Two design laws bent under real pressure. "No modals" was one of the explicit non-negotiables, and the build now has three overlay-style surfaces that weren't in the original spec: an in-terminal file editor (arguably in-bounds, since it's how conflict resolution has to work without a real `$EDITOR`), a cookie-consent card, and an info/about panel. The consent banner exists because analytics got added post-launch and needed a gate; the info panel exists because the tool needed a place to explain itself and credit its author. Both are dismissible and don't block gameplay, but they are floating cards with backdrops, which is exactly the pattern the plan said to avoid. The other bend is "no sidebar" - the plan explicitly called for a single centered column with no sidebar chrome, and the build now has two persistent side panels (a Commands reference and a Tips list) because early testing made clear that a first-time player doesn't necessarily know basic Git syntax and "discoverable by typing" wasn't enough on its own. Both panels are collapsible and off by default would arguably fit the original spirit better, but they currently default to open.

## What's still outstanding

A few things from the plan never got built. "Current streak" was named in the design law's local-stats bullet and never implemented - only per-level completion and time are stored, no streak counter. The plan's auto-advance was supposed to be skippable by pressing any key; right now it's a fixed ~900ms delay with no skip. "Median attempts per level" and "time-to-first-command," both named as MVP success metrics, aren't directly instrumented - hint usage and command-error events are a reasonable proxy for the first, but neither is tracked as its own event yet.

The bigger open item is testing. The sandboxed shell needed to actually run this code was down for the entire build, so nothing has been executed in an automated way - no unit tests on the engine, no headless run of the ten levels. Confidence comes entirely from manually tracing each level's logic against the code and from a handful of real playtest rounds that already caught and fixed real issues (a `git --help` dead-end, a branch-name mismatch that looked like a bug but wasn't, a Mac keyboard shortcut gap). That loop has worked, but it means there are almost certainly untested paths, especially alternate valid solutions to levels 6 through 10 that haven't been tried by a live player yet. A proper playtest pass - someone sitting down and clearking through all ten levels a couple of different ways - should happen before this gets called stable.

Two more loose ends: mobile/responsive layout was explicitly deprioritized in the plan and still hasn't been tested at all, and it matters more now than it did at plan time, since the side panels, the centered objective bar, and the fixed-width cookie card all assume a reasonably wide viewport. And the consent banner's legal correctness for GDPR/UK audiences was flagged earlier as something worth a real legal look, not just an engineering guess - that's still unresolved.

## Extra work beyond the original plan

Several things got added in direct response to real friction, not from the original spec. A persistent OBJECTIVE bar above the input now keeps the current goal visible at all times, added after a player did everything right except use the exact branch name the objective asked for and had no way to recheck it without scrolling. The input box grew from a slim bottom bar into a larger, centered, panel-styled focus area closer to Monkeytype's actual typing-test prominence. A one-time first-visit intro and input placeholder text now do some of the onboarding work the plan assumed "discoverable by typing" would cover on its own.

The analytics stack ended up considerably ahead of what the plan asked for. The plan wanted local-only stats; the build now has GA4 with custom events for level starts, level completions (with time and hint count), hint requests, resets, campaign completion, and both git and non-git command errors - enough to actually answer the plan's own success-metric questions from a dashboard instead of guessing. Vercel Web Analytics and Microsoft Clarity session replay are wired in alongside it, all gated behind the cookie-consent decision so nothing fires until a visitor opts in.

A few things exist purely for growth and credibility: a GitHub badge in the title bar with a live, gold-highlighted star count pulled from the repo's API, and the info panel doubling as a lightweight about/credits section with a real `<meta name="description">` and Open Graph tags for actual search and link-preview benefit. A small level-up flourish - the level indicator pulses and the terminal edge glows briefly in the current accent color - now marks the exact moment a level completes, beyond the plan's simpler "brief flash."

## Honest read on where this stands

The core thesis - realistic Git commands, solved through any valid sequence, in a fast terminal-styled loop - works and matches the original plan closely enough that the plan held up well as a spec. The places it drifted (the engine, the side panels, the consent/info overlays) were each a reasonable trade made under a real constraint, not scope creep for its own sake, but they're worth a deliberate look later rather than treating as permanent: in particular, whether the side panels should default to collapsed, and whether the consent/info cards can be reskinned to feel less like a "website" and more like the terminal they're floating on top of.

The single biggest risk right now isn't a missing feature, it's the testing gap. Everything has been reasoned through carefully, but "carefully reasoned" and "actually run" are not the same guarantee, and the bugs found so far were all found by a person playing the game, not by any verification step on this end. Before calling this done, it's worth a dedicated pass clicking through all ten levels - including a couple of alternate, non-obvious solution paths per level - on the actual deployed Vercel URL, on both desktop and a narrow viewport, to find whatever the manual trace missed.
